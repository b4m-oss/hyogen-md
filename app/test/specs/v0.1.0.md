# v0.1.0 テスト仕様書

[roadmap v0.1.0](../../docs/roadmap.md) の実装範囲に対応する。製品仕様は [docs/specs/](../../docs/specs/) が正。

## スコープ

- `renderServer(source: string, context?, options?)`（文字列入力のみ）
- front matter、本文 `{{ }}`（最小式 + デフォルトパイプ）、`include`、`strip` オプション
- エラー: `parse_error` / `file_not_found` / `frontmatter_too_large` / `load_failed`

## 確定事項（v0.1.0）

| 項目 | 確定内容 |
|------|----------|
| 子 include の再帰 | 子ドキュメントにも前段パイプラインを **再帰適用** してから親に埋め込む |
| 未知 `code` の formatMessage | **空文字**（v0.1.0 実装で固定） |
| `root` 未指定 + 相対 include | **`parse_error`** |
| `@endhg` 欠落 | scan 段では未検出。**execute 段** で **`parse_error`** |
| オブジェクトの `{{ }}` 展開 | **`String(value)` 相当**（例: `[object Object]`） |

### 実装で追加固定した挙動（v0.1.0）

| 項目 | 挙動 |
|------|------|
| `createHyogenError` の空 `code` | 空のまま（フォールバックなし） |
| `mergeContext` の `null` / `undefined` 引数 | スキップ |
| `preserveHgComments: true` | include 展開後、元 hg コメントを展開内容の直前に再挿入 |

## Phase 0: 基盤

### formatMessage

- [messages.en.json](../../docs/specs/messages.en.json) のテンプレート文字列とプレースホルダ値を受け取り、展開済みメッセージを返す
- プレースホルダは `{key}` 形式。未知キーはそのまま残す
- 副作用なし（純粋関数）

#### テスト: 正常系

- `code: "file_not_found"`, `details: { path: "./a.md", from: "./b.md", via: "include" }` のとき、`File not found: ./a.md (referenced from ./b.md via include)` を返す
- プレースホルダが 1 つだけのテンプレートでも正しく置換される（例: `parse_error` の `{message}`）
- 値に空文字を渡した場合も、そのまま空文字として展開される

#### テスト: 異常系

- 存在しない `code` を渡したとき、**空文字**を返す
- テンプレートに `{` だけがあり閉じ `{` がない場合、例外を投げずに可能な限り展開する（または未展開のまま返す — 実装方針をテストで固定）
- `details` が `undefined` のとき、プレースホルダは `{key}` のまま残る

### createHyogenError

- `code` / `message` / 任意の `path` / `details` から `HyogenError` を生成する
- 生成物は `Error` を継承し、`code` / `message` / `path` / `details` プロパティを持つ（[api.md](../../docs/specs/api.md)）
- `message` 未指定時は `formatMessage(code, details)` で英語メッセージを補完する

#### テスト: 正常系

- `code: "parse_error"`, `path: "page.md"`, `details: { message: "unexpected token" }` で、`error.code === "parse_error"` かつ `error.message` に `unexpected token` が含まれる
- `message` を明示した場合、その文字列が `error.message` になる（formatMessage は呼ばない）
- `instanceof Error` が `true` である

#### テスト: 異常系

- `code` が空文字のとき、生成自体はするが `code` が空のまま（または `parse_error` にフォールバック — 実装方針をテストで固定）
- `details` に循環参照オブジェクトを渡しても JSON 化時に落ちない（メッセージ生成が例外を投げない）
- 未知 `code` で `message` も未指定のとき、**空文字**を返す

## Phase 1: Front matter + context

### parseFrontMatter

- ソース文字列先頭の YAML front matter（`---` 囲み）を切り出してパースする
- パース結果を context 用オブジェクトとして返す。本文は front matter 除去後の文字列
- front matter ソース（生テキスト）のバイト長が 64KB を超えると `frontmatter_too_large`（[variables.md](../../docs/specs/variables.md)）

戻り値（想定）:

```ts
{
  context: Record<string, unknown>;
  body: string;
  rawFrontMatter?: string;  // preserveFrontMatter 用
}
```

#### テスト: 正常系

- `---\ntitle: Hello\n---\n# Body` → `context: { title: "Hello" }`, `body: "# Body"`, `rawFrontMatter` に YAML 部分
- front matter が無いとき → `context: {}`, `body` はソース全体, `rawFrontMatter` は undefined
- 複数行 YAML（ネストしたオブジェクト）を正しく読み取れる

#### テスト: 異常系

- front matter ソースが 64KB を超える → `frontmatter_too_large`
- `---` のみで YAML 本体が無い（`---\n---\n`）→ `parse_error`
- 不正な YAML 構文 → `parse_error`

### mergeContext

- 複数の context オブジェクトを **後勝ち** で浅いマージする（[variables.md](../../docs/specs/variables.md)）
- 同名キーは後から渡したオブジェクトの値で上書きする
- 副作用なし。入力オブジェクトは変更しない

#### テスト: 正常系

- `{ a: 1 }` と `{ b: 2 }` をマージすると `{ a: 1, b: 2 }` になる
- `{ title: "old" }` の後に `{ title: "new" }` をマージすると `{ title: "new" }` になる
- 引数 0 個または `undefined` のみのとき `{}` 相当になる

#### テスト: 異常系

- 引数に `null` を含めた場合の扱いを固定する（スキップするか空オブジェクト扱い — テストで固定）
- ネストオブジェクトは浅いマージのみ（`{ meta: { x: 1 } }` + `{ meta: { y: 2 } }` → 後者で `meta` 全体が置換）
- 配列値は上書きのみ（マージしない）

## Phase 2: hyogen ブロック

### scanHgBlocks

- Markdown ソースから hyogen HTML コメントブロック（`@hg` … `@endhg`）を **出現順** に列挙する
- 各要素に `{ start, end, inner, raw }` 相当の位置情報と内部テキストを返す
- コードフェンス（`` ``` ``）内の HTML コメントは **無視** する（[dsl.md](../../docs/specs/dsl.md)）

#### テスト: 正常系

- 改行あり形式 `<!--\n@hg\ninclude ./a.md\n@endhg\n-->` を 1 ブロックとして検出する
- 同一行 minify 形式 `<!--@hg\ninclude ./a.md\n@endhg-->` を 1 ブロックとして検出する
- 複数ブロックがあるとき、文書出現順の配列になる

#### テスト: 異常系

- `@hg` のみで `@endhg` が無いとき、scan 段では未検出。**execute 段** で **`parse_error`**
- 通常本文中の `<!-- 通常コメント -->` は hyogen ブロックとして検出しない
- コードフェンス内の `<!-- @hg ... @endhg -->` は検出しない

### parseIncludeDirective

- hyogen ブロック内部テキストから `include <path>` 指令を解析する
- v0.1 では **include 1 行のみ** 許可。前後の空白・改行は許容
- 解析結果は `{ kind: "include", path: string }`。include 以外の文は v0.1 では拒否

#### テスト: 正常系

- 入力 `include ./partials/description.md` → `{ kind: "include", path: "./partials/description.md" }`
- 前後に空行があっても path を正しく取り出す
- パスに `./` や `../` を含む相対パスをそのまま返す（正規化は `resolveIncludePath` 側）

#### テスト: 異常系

- `include` のみで path が無い → `parse_error`
- `include ./a.md` と `const x = 1` が同一ブロックにある → `parse_error`（v0.1 は include 単独行のみ）
- `extend ./layout.md` 等、include 以外の指令 → `parse_error`
- 空文字列 → `parse_error`

### executeHgBlocks

- 文書中の hyogen ブロックを文書順に処理し、v0.1 では **include 指令の収集** のみ行う
- include ブロックの位置を展開用マーカー（またはプレースホルダ）に置き換えたソースを返す
- 収集した include 指令リストを返す（path と元位置）

#### テスト: 正常系

- include ブロック 1 つ → 指令 1 件が収集され、該当コメント位置がマーカーに置換される
- include ブロックが無いソース → 指令 0 件、ソースは実質不変
- 複数 include → 出現順に指令が並ぶ

#### テスト: 異常系

- include 以外の hyogen 文（`const x = 1`）→ `parse_error`
- `@endhg` 欠落ブロック → **execute 段** で `parse_error`（scan 段では未検出）
- 同一ブロック内に include が 2 行 → `parse_error`（v0.1）

## Phase 3: 式エンジン

### parseExpression

- `{{ }}` 内の式文字列を AST（または評価可能な中間表現）にパースする
- v0.1 許可: 識別子、文字列/数値/真偽/null/undefined リテラル、`.` メンバアクセス、`|` デフォルトパイプ
- 任意 JS 評価は行わない。ホワイトリスト外は `parse_error`

#### テスト: 正常系

- `title` → 変数参照ノード
- `object.key` → メンバアクセスノード
- `color | "transparent"` → デフォルトパイプノード（左: 識別子、右: 文字列リテラル）

#### テスト: 異常系

- `foo()` → `parse_error`（関数呼び出し不可）
- `a ? b : c` → `parse_error`（三項不可）
- `arr[0]` → `parse_error`（添字不可。v0.1 は `.` のみ）
- `1 + 2` → `parse_error`（算術演算不可）
- 空文字 → `parse_error`

### evaluateExpression

- `parseExpression` の結果と context を受け取り、JavaScript 値を返す
- 未束縛識別子は `undefined`（[variables.md](../../docs/specs/variables.md)）
- デフォルトパイプは **JS `||` 相当** の falsy 判定（`undefined` / `null` / `""` / `0` / `false` 等で右辺）

#### テスト: 正常系

- context `{ title: "Hello" }` + 式 `title` → `"Hello"`
- context `{ meta: { id: 1 } }` + 式 `meta.id` → `1`
- context `{}` + 式 `color | "transparent"` → `"transparent"`（未束縛は falsy）

#### テスト: 異常系

- context `{ count: 0 }` + 式 `count | "none"` → `"none"`（0 は falsy）
- context `{ flag: false }` + 式 `flag | "yes"` → `"yes"`
- 存在しない中間プロパティ `a.b.c`（`a` が undefined）→ エラーにせず `undefined`（メンバアクセスの慣例）
- v0.1 では `forbidden_property_access` は **未実装**（`__proto__` 参照は v0.2）。テストは「エラーにしない」で固定

### interpolateExpressions

- Markdown 本文中の `{{ expr }}` および `{{{ expr }}}` を検出し、評価結果の文字列で置換する
- v0.1 では `{{ }}` と `{{{ }}}` の **挙動は同一**（エスケープなし。[variables.md](../../docs/specs/variables.md)）
- 置換後の Markdown 文字列を返す。副作用なし

#### テスト: 正常系

- 本文 `Hello {{ name }}` + context `{ name: "World" }` → `Hello World`
- `{{ title | "Untitled" }}` で context に `title` が無いとき → デフォルト値が埋まる
- 同一行に複数の `{{ }}` があるとき、すべて置換される

#### テスト: 異常系

- `{{ foo() }}` → `parse_error`（式パース段階）
- 閉じ `}}` が無い `{{ name` → `parse_error`
- 値がオブジェクトのとき **`String(value)` 相当**（`[object Object]`）で埋まる

## Phase 4: include

### resolveIncludePath

- `root` ディレクトリと include 指令の相対 path から、読み込み用の絶対パスを解決する
- v0.1: `path.resolve(root, includePath)` 相当。`.doc_root` 走査は **未実装**（v0.2）
- `root` 未指定で相対 path のときは **`parse_error`**

#### テスト: 正常系

- `root: "/project/docs"`, `includePath: "./partials/a.md"` → `"/project/docs/partials/a.md"`（OS 区切りは正規化）
- `root: "/project/docs"`, `includePath: "../other/b.md"` → `"/project/other/b.md"`（`../` 許容。[paths.md](../../docs/specs/paths.md) doc_root 無し時）
- `root` が trailing slash 付きでも正しく解決する

#### テスト: 異常系

- `root` が `undefined` / 空で相対 path → `parse_error`
- `includePath` が空文字 → `parse_error`
- v0.1 では絶対 path の doc_root 内チェックは **未実装** — 絶対 path 入力時はそのまま resolve

### createFsLoader

- Node.js FS からファイルを読み込む `Loader` 関数 `(path: string) => Promise<string>` を生成する
- 成功時は UTF-8 文字列を返す
- 失敗時は [api.md](../../docs/specs/api.md) に従い `file_not_found` または `load_failed` の `HyogenError` を投げる

#### テスト: 正常系

- 存在するファイルパス → ファイル内容文字列
- 空ファイル → `""`
- UTF-8 日本語を含むファイル → 文字化けせず読める

#### テスト: 異常系

- 存在しないパス → `file_not_found`（`details.path` / `details.from` / `details.via` を含めうる）
- ディレクトリパスを渡した → `load_failed`
- 読み取り権限が無いファイル → `load_failed`（OS 依存。可能なら skip または mock）

### expandIncludes

- マーカー付きソースと include 指令リストを受け取り、各 include 先の内容で置換する
- 読み込んだ子ドキュメントに対しても **再帰的** に前段パイプライン（front matter 除去 → hg 実行 → 子 include 展開 → …）を適用してから親に埋め込む
- v0.1 では **循環 include 検出なし**（v0.2 で `circular_include` 警告）

#### テスト: 正常系

- 親 + include `./partial.md`（内容 `Partial body`）→ `Partial body` に置換
- ネスト include（A → B → C）→ 最終的に C の内容まで展開
- include 先の `@hg include` も子の再帰処理で展開される

#### テスト: 異常系

- include 先が存在しない → `file_not_found` で中断
- `root` 未設定で相対 include → `parse_error`
- loader が reject → `load_failed`

## Phase 5: strip + オーケストレータ

### stripHgComments

- ソースから hyogen HTML コメントブロック（`@hg` … `@endhg` を含む `<!-- ... -->`）を除去する
- `preserveHgComments: true` のときはソースを変更せず返す（[api.md](../../docs/specs/api.md)）
- include 展開**後**に実行する想定（step 7）。展開済み本文は残す

#### テスト: 正常系

- `Hello\n<!--\n@hg\ninclude ./a.md\n@endhg\n-->\nWorld` → `Hello\n\nWorld`（ブロック除去）
- `preserveHgComments: true` → 入力と同一
- hyogen ブロックが無いソース → 不変

#### テスト: 異常系

- 不完全な `<!-- @hg` のみ → 除去しない（そのまま残す）
- 通常 HTML コメント `<!-- note -->` は **除去しない**（hyogen 標識を含むもののみ）
- 空文字 → 空文字

### applyFrontMatterOutputOption

- レンダリング結果に対し、front matter の出力可否を適用する
- デフォルト（`preserveFrontMatter: false`）: 出力から front matter を除去（[api.md](../../docs/specs/api.md)）
- `preserveFrontMatter: true`: 元の front matter ブロック（`---` 囲み）を出力先頭に残す

#### テスト: 正常系

- front matter 付き入力 + `preserveFrontMatter: false` → 出力に `---` ブロックが無い
- `preserveFrontMatter: true` → 出力先頭に元の YAML front matter が復元される
- front matter 無し入力 → オプションに関わらず不変

#### テスト: 異常系

- front matter ソースは保持しているが YAML が壊れていた場合 → パース段階で既に `parse_error`。本関数には到達しない
- `preserveFrontMatter: true` だが `rawFrontMatter` が無い → 出力に front matter 無し

### renderDocument

- 1 本の Markdown ソースと context / options を受け取り、パイプライン全体を実行する
- 戻り値は `{ markdown: string, warnings: HyogenWarning[] }`（v0.1 では `warnings` は常に空配列）
- 処理順: front matter → hg 実行 → include 展開 → `{{ }}` 評価 → strip（[pipeline.md](../../docs/specs/pipeline.md) の v0.1 有効段）

#### テスト: 正常系

- front matter `{ title: "T" }` + 本文 `# {{ title }}` → `# T`
- include + 変数展開が組み合わさった fixture で期待 Markdown が得られる
- `preserveHgComments: true` / `preserveFrontMatter: true` が options 経由で効く

#### テスト: 異常系

- 不正 YAML front matter → `frontmatter_too_large` または `parse_error` で throw
- include 先欠落 → `file_not_found` で throw
- 不正 `{{ expr }}` → `parse_error` で throw

### renderServer

- 公開 API。`renderServer(source: string, context?, options?)` で Node SSR/SSG 向けに 1 本を描画する（[api.md](../../docs/specs/api.md)）
- v0.1 では **`source` は文字列のみ**（`{ path }` 形式は v0.2）
- `options.root` を `renderDocument` / include 解決に渡す。`options.serverContext` は v0.1 では `context` とマージしてよい（厳密分離は v0.5）

#### テスト: 正常系

- 最小ソース `"# Hello"` → `{ markdown: "# Hello", warnings: [] }`
- `context` + front matter + 本文変数が後勝ちで反映される
- `options.root` 指定時、相対 include が解決される

#### テスト: 異常系

- `source` が空文字 → 空 markdown
- include に `root` 無し → `parse_error`
- 内部 `renderDocument` が throw した `HyogenError` をそのまま reject する

## 統合シナリオ（end-to-end）

[roadmap v0.1.0](../../docs/roadmap.md) の「テスト」節 4 項目。`renderServer` + fixture で検証。

#### テスト: 正常系

- **front matter → 変数展開**: `---\ntitle: My Page\n---\n# {{ title }}` → `# My Page`
- **include 単純埋め込み**: 親 md が `@hg include ./partials/body.md @endhg`、`options.root` 指定 → partial 本文が展開される
- **デフォルトパイプ**: 本文 `{{ missing | "fallback" }}` → `fallback`

#### テスト: 異常系

- **不正 DSL → parse_error**: hyogen ブロック内 `const x = 1` → `parse_error`
- include 先が存在しない → `file_not_found`
- front matter が 64KB 超 → `frontmatter_too_large`
