# v0.11.0 テスト仕様書

[roadmap v0.11.0](../../dev-docs/roadmap.md) の実装範囲に対応する。製品仕様は [dev-docs/specs/](../../dev-docs/specs/) が正。

## スコープ

- **`loadDataSources(sources, options?)`** — 複数データファイル（YAML / JSON / CSV）を読み、変数名 → 値の `HyogenContext` を返す（[api.md](../../dev-docs/specs/api.md)）
- **`parseDataFile(source, path)`** — 拡張子に応じて JSON / YAML / CSV をパース（内部 API。テスト対象）
- **`parseCsvFromString(source)`** — CSV 文字列 → オブジェクト配列（`csv-parser` + `Readable.from` の薄いラッパ。内部 API。テスト対象）
- **`dataSources` オプション** — `renderServer` / `build` で読込結果を context にマージ
- **マージ順** — `dataSources` → `context` → `serverContext`（浅いマージ、後勝ち）
- **`renderClient` 拒否** — `dataSources` 指定時は **`data_sources_on_client`**
- v0.10.0 回帰（470+ テスト）維持

## スコープ外（v0.11.0）

- DSL からの `import` / `require`（引き続き禁止）
- `renderClient` / `@b4moss/hyogen-md/client` への `loadDataSources` export
- データファイルのリモート URL 読込
- Playground のデータ読込 UI
- TOC / 許可メソッド拡張（v0.12+）
- CSV の TSV・セミコロン区切り・BOM 除去（将来 PATCH で可）

## Phase 1: parseCsvFromString

新規 `app/src/io/parseCsvFromString.ts`

### parseCsvFromString

- CSV 文字列を `Readable.from` → **`csv-parser`** でパースし、ヘッダー行 + データ行からオブジェクト配列を返す
- フィールドはすべて **文字列**（型推論はしない）
- 空行はスキップ
- 列数不一致は **`parse_error`**

#### テスト: 正常系

- `id,name\n1,Alpha\n2,Beta` → `[{ id: "1", name: "Alpha" }, { id: "2", name: "Beta" }]`
- `rows-quoted.csv`（`"a,b",c` 形式）→ クォート内カンマは 1 フィールド
- `\r\n` 改行の CSV も同様にパース
- ヘッダー 1 列・データ 0 行 → `[]`

#### テスト: 異常系

- データ行の列数がヘッダーと不一致 → **`parse_error`**
- 空文字列 → **`parse_error`**
- ヘッダー行のみ（改行なしデータなし）かつ空でないヘッダー → `[]`（データ 0 行は正常）

## Phase 2: parseDataFile

新規 `app/src/io/parseDataFile.ts`

### parseDataFile

- `(source: string, path: string)` で拡張子を判定しパーサへ委譲
- 戻り値は `unknown`（呼び出し側が context 値として格納）

#### テスト: 正常系

- `site.yaml` 相当 → オブジェクト（`title` キー等）
- `products.json` → オブジェクト
- `items-array.json` → トップレベル配列 `[ "a", "b" ]`
- `rows.csv` → オブジェクト配列
- 大文字小文字: `.YAML` / `.JSON` / `.CSV` も拡張子判定で対応（path の末尾で判定）

#### テスト: 異常系

- `invalid.json` → **`parse_error`**
- `invalid.yaml` → **`parse_error`**
- `invalid.csv` → **`parse_error`**
- `unknown.txt` → **`parse_error`**（`details.format: "txt"`）
- `empty.json`（空ファイル）→ **`parse_error`**

## Phase 3: loadDataSources

新規 `app/src/io/loadDataSources.ts`

### loadDataSources

- `DataSourcesMap` の各エントリを順に読込・パースし、1 つの `HyogenContext` にまとめる
- キー = 変数名、値 = ルート相対パス
- `options.root` 未指定時は cwd または `.doc_root` 探索（既存 path 解決に合わせる）
- `options.loader` 指定時は loader で文字列取得（テストでは FS スタブ可）

#### テスト: 正常系

- `{ site: "./data/site.yaml" }` → `{ site: { title: "...", ... } }`
- 複数キー `{ site: "...", products: "..." }` → 両方のキーが context に存在
- `items-array.json` を `items` キーで読込 → `context.items` が配列
- `rows.csv` を `rows` キーで読込 → `context.rows` がオブジェクト配列
- 空マップ `{}` → `{}`

#### テスト: 異常系

- 存在しないパス → **`file_not_found`**（`details` に要求パス）
- パース失敗 → **`parse_error`**（読めたが中身が壊れている）
- loader が throw → **`load_failed`**

## Phase 4: renderServer 統合

変更: [renderServer.ts](../../src/renderServer.ts)

### renderServer（dataSources）

- `options.dataSources` 指定時、レンダリング前に `loadDataSources` を実行し context にマージ

#### テスト: 正常系

- `from-yaml.md` + `dataSources: { site: "./data/site.yaml" }` → 本文 `{{ site.title }}` が展開
- `from-json.md` + `dataSources: { meta: "./data/meta.json" }` → `{{ meta.version }}` 展開
- `from-csv.md` + `dataSources: { rows: "./data/rows.csv" }` → `{{ rows.0.name }}` 展開
- `merge-order.md` + `dataSources` と `context` に同名キー → **context の値**が勝つ
- `dataSources` + `serverContext` に同名キー → **serverContext の値**が勝つ
- `dataSources` 未指定 → v0.10.0 回帰（従来どおり）

#### テスト: 異常系

- 欠落 data ファイル → **`file_not_found`**（render 中断）
- 壊れた JSON → **`parse_error`**

## Phase 5: build 統合

変更: [build/build.ts](../../src/build/build.ts)

### build（dataSources）

- エントリ収集・依存検証の後、各エントリ render の前に `loadDataSources` を 1 回実行

#### テスト: 正常系

- `build({ input: "pages/**/*.md", outDir, dataSources: { config: "./data/config.yaml" } })` → 出力 Markdown に `{{ config.env }}` が展開
- 複数エントリでも **同一** data 値が使われる
- `dataSources` 未指定 → v0.10.0 build 回帰

#### テスト: 異常系

- 欠落 data → **`file_not_found`**（build 中断、ファイル未書き出し）
