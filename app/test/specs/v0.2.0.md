# v0.2.0 テスト仕様書

[roadmap v0.2.0](../../docs/roadmap.md) の実装範囲に対応する。製品仕様は [docs/specs/](../../docs/specs/) が正。

## スコープ

- **Phase 0（基盤）**: Vite ライブラリモード + Vitest（v0.1 回帰 Green）
- `renderServer(source: string | { path: string }, context?, options?)`
- `.doc_root` 基準のパス解決（[paths.md](../../docs/specs/paths.md)）
- 循環 include → `circular_include` **警告**（スキップ、中断しない）
- 危険キー → `forbidden_property_access` **エラー**（[security.md](../../docs/specs/security.md)）
- `component path as name` 登録 + `{{ name({ ... }) }}` 呼び出し
- props 契約（front matter `props:`）+ `prop_*` **警告**
- エイリアス衝突 → `duplicate_component_alias` / `alias_collision` **エラー**
- component 出力は **単行のみ**（`component_multiline_output` エラー）

## Phase 1: パス解決 + エントリ

### findDocRoot

- 開始ディレクトリから祖先方向へ `.doc_root` マーカーファイルを探索する
- 見つかったディレクトリを **`rootDir`** として返す
- 見つからない場合は `undefined`（または null — 実装方針をテストで固定）
- マーカーの中身は評価しない（存在のみ）

#### テスト: 正常系

- `doc-root/pages/index.md` の親階層に `.doc_root` がある → `rootDir` が `doc-root/` に相当
- 開始ディレクトリ自身が `.doc_root` を含む → そのディレクトリが `rootDir`
- 深いネスト（`doc-root/pages/nested/deep.md` の dir）からも同じ `rootDir` を返す

#### テスト: 異常系

- `.doc_root` が存在しないツリー → `undefined`
- 開始パスがファイルの場合、その親ディレクトリから探索する
- ファイルシステムルートまで到達しても見つからない → `undefined`

### resolveTemplatePath

- include / component 用のパスを解決する（[paths.md](../../docs/specs/paths.md)）
- `./` `../` 相対、`/` ルート相対（`.doc_root` あり時のみ）、絶対パス（正規化後 rootDir 内のみ）に対応
- `rootDir` と `fromPath`（参照元ファイルの絶対パス）を受け取る

#### テスト: 正常系

- `.doc_root` あり + `/partials/header.md` → `rootDir/partials/header.md` に解決
- 相対 `./partials/body.md` + `fromPath` が `pages/index.md` → 正しい絶対パス
- `../` による兄弟ディレクトリ参照が rootDir 内なら解決される

#### テスト: 異常系

- `.doc_root` 無し + ルート相対 `/foo.md` → **`parse_error`**
- 解決後の実体が rootDir 外 → **`parse_error`** または **`file_not_found`**（実装方針をテストで固定）
- 存在しないパス → **`file_not_found`**
- 空文字パス → **`parse_error`**

### assertWithinRootDir

- 正規化 + symlink 追従後、実体パスが `rootDir` 配下か検証する
- rootDir 外なら **`parse_error`**（または専用 code — 実装方針をテストで固定）

#### テスト: 正常系

- rootDir 内の通常ファイル → 例外なし
- rootDir 内の symlink 先（実体も rootDir 内）→ 例外なし

#### テスト: 異常系

- rootDir 外への symlink 追従 → 拒否
- `../` で rootDir 外に出る正規化結果 → 拒否
- rootDir 外の絶対パス → 拒否
- 存在しないパス → **`file_not_found`**

### resolveRenderInput

- `renderServer` の第 1 引数を正規化する
- `string` → そのままソース文字列として `{ source, rootDir? }` を返す
- `{ path: string }` → FS 読込 + `path` のディレクトリから `findDocRoot` → `{ source, rootDir, entryPath }`

#### テスト: 正常系

- 文字列入力 → `{ source: 入力文字列 }`（rootDir 未設定）
- `{ path: "doc-root/pages/index.md" }` → ファイル内容 + `rootDir` が doc-root
- `options.root` 指定時 → doc_root 探索の起点/上書きに使用

#### テスト: 異常系

- 存在しない `{ path }` → **`file_not_found`**
- ディレクトリを `{ path }` に指定 → **`parse_error`** または **`load_failed`**
- 読込権限なし → **`load_failed`**

### renderServer（overload 拡張）

- v0.1 の `string` 入力に加え `{ path: string }` を受け付ける（[api.md](../../docs/specs/api.md)）
- `{ path }` 時は `resolveRenderInput` → `renderDocument` に `rootDir` を伝播
- 戻り値 `{ markdown, warnings }` の `warnings` をパイプラインから集約

#### テスト: 正常系

- 既存 v0.1 文字列入力テストが引き続き Green
- `{ path: "doc-root/pages/index.md" }` で Markdown が返る
- ルート相対 include `/partials/body.md` が `{ path }` エントリから解決される

#### テスト: 異常系

- `{ path }` + include 先が rootDir 外 → エラー
- `.doc_root` 無し + `{ path }` + ルート相対 include → **`parse_error`**
- 内部 `HyogenError` をそのまま reject

## Phase 2: 循環 include + 警告基盤

### VisitStack / checkCircularInclude

- 正規化絶対パス（または同等キー）の Set で訪問履歴を管理
- 再訪問検出時 `{ circular: true, path, from }` を返す

#### テスト: 正常系

- 初回訪問 → circular ではない
- 異なるファイルの連続 visit → circular ではない
- visit 後に pop（スコープ終了）→ 再 visit 可（include ネスト終了時）

#### テスト: 異常系

- 同一ファイルへの再 visit（A→B→A）→ circular
- 自己参照 include → circular
- 空パス → 実装方針をテストで固定

### expandIncludes（循環拡張）

- v0.1 挙動に加え VisitStack で循環検出
- 循環時 **`circular_include` 警告** + 当該 include **スキップ**（中断しない）
- 警告は `RenderResult.warnings` に集約

#### テスト: 正常系

- 非循環 include は v0.1 どおり展開
- A→B→A で 2 周目 A の include がスキップされ、markdown は部分展開
- 警告配列に `code: "circular_include"` が含まれる

#### テスト: 異常系

- include 先欠落は従来どおり **`file_not_found`**（循環とは別）
- 循環検出後も他の include / 式評価は継続
- 深い非循環ネストも正常展開

## Phase 3: 危険キー

### isForbiddenKey

- 識別子が危険キーリストに含まれるか判定（[security.md](../../docs/specs/security.md)）
- 対象: `__proto__`, `prototype`, `constructor`, `__defineGetter__`

#### テスト: 正常系

- 通常識別子 `title`, `userName` → false
- 大文字小文字区別（`__PROTO__` 等）→ 実装方針をテストで固定

#### テスト: 異常系

- `__proto__` → true
- `prototype` → true
- `constructor` → true
- `__defineGetter__` → true

### assertSafeMemberAccess

- member access / identifier 解決前に `isForbiddenKey` を適用
- 危険キーなら **`forbidden_property_access`** を throw

#### テスト: 正常系

- `obj.title` で `title` が安全 → 例外なし
- リテラル・デフォルトパイプは従来どおり

#### テスト: 異常系

- `obj.__proto__` → **`forbidden_property_access`**
- `obj.prototype` → **`forbidden_property_access`**
- ネスト `a.b.__proto__` → **`forbidden_property_access`**
- context 直下 `{{ __proto__ }}` → **`forbidden_property_access`**

### evaluateExpression（async 化 + 危険キー）

- v0.1 同期評価を **async** に拡張（component 呼び出し対応）
- member access / identifier 解決時に `assertSafeMemberAccess`
- v0.1「`__proto__` をエラーにしない」テストは v0.2 仕様に **更新**

#### テスト: 正常系

- 既存 v0.1 式評価（変数・リテラル・メンバ・デフォルトパイプ）が Green
- `.toLocaleString('ja-JP')` が数値に適用できる

#### テスト: 異常系

- `__proto__` アクセス → **`forbidden_property_access`**
- 未登録関数呼び出し → **`parse_error`**
- 不正構文 → **`parse_error`**

## Phase 4: component 登録

### parseComponentDirective

- hyogen ブロック内の `component <path> as <alias>` を字句解析
- path は include と同じ形式（クォート有無 — 実装方針をテストで固定）
- alias は JS 識別子相当

#### テスト: 正常系

- `component ./components/greeting.md as greet` → `{ path: "./components/greeting.md", alias: "greet" }`
- ルート相対 `/components/greeting.md as greet` を受理（doc_root あり時）
- 前後空白を trim

#### テスト: 異常系

- `component` のみ / `as` 欠落 → **`parse_error`**
- 不正 alias（数字始まり等）→ **`parse_error`**
- `include` と同一行に共存 → **`parse_error`**
- 空 path → **`parse_error`**

### ComponentRegistry

- 1 レンダリングセッション中の component 登録を保持
- `register({ alias, componentPath, definedAt })` / `get(alias)` / `has(alias)`

#### テスト: 正常系

- 登録後 `get("greet")` で定義が返る
- 複数 alias を別名で登録可能
- include 子ドキュメントから登録した alias も同一 registry で見える

#### テスト: 異常系

- 同一ファイルで同名 `as` 二重 → **`duplicate_component_alias`**
- 既存 context キーと alias 衝突 → **`alias_collision`**
- 親で登録済み alias を子で再登録 → **`alias_collision`**
- 未登録 alias の `get` → `undefined`

### executeHgBlocks（拡張）

- v0.1 の `include` に加え `component … as` を文書順に処理
- component ブロックは **マーカー除去** + **レジストリ登録**（strip 対象から除去）
- `include` と `component` を **同一ブロックに共存不可**

#### テスト: 正常系

- `include` のみ → v0.1 どおり
- `component … as` のみ → 登録 + ブロック除去
- include 先で登録した component もレジストリに反映

#### テスト: 異常系

- 同一ブロックに include + component → **`parse_error`**
- 不正 component 行 → **`parse_error`**
- `@endhg` 欠落 → **`parse_error`**
- alias 衝突 → **`duplicate_component_alias`** / **`alias_collision`**

## Phase 5: 式エンジン + component 呼び出し

### parseExpression（call / object literal / toLocaleString）

- v0.1 subset に加え:
  - `identifier({ key: value, ... })` — object literal（string/number/boolean/null のみ v0.2）
  - `expr.toLocaleString('ja-JP')` — 1 メソッドのみ

#### テスト: 正常系

- `greet({ name: "Ada" })` → call ノード + props オブジェクト
- `{ city: "Osaka", count: 1 }` リテラル各型
- `population.toLocaleString('ja-JP')` → method call ノード
- 既存 v0.1 式（変数・リテラル・メンバ・パイプ）が Green

#### テスト: 異常系

- 未対応メソッド `.toFixed()` → **`parse_error`**
- object literal 内に式・変数参照 → **`parse_error`**（v0.2）
- 空引数 `greet()` → 実装方針をテストで固定
- 不正 object literal 構文 → **`parse_error`**

### parsePropsContract

- component ファイル front matter の `props:` マップをパース（[templating.md](../../docs/specs/templating.md)）
- 各 prop: `type`, `isRequired`, `default`

#### テスト: 正常系

- 正式 YAML `props: { name: { type: string, isRequired: true } }` → 契約オブジェクト
- `type` 省略 → 推論用に undefined type
- props 節無し → 空契約（検証緩い）

#### テスト: 異常系

- 不正 YAML → **`parse_error`**
- 未知 `type` 値 → warning または parse_error（実装方針をテストで固定）
- props が配列 → **`parse_error`**

### validateProps

- 呼び出し側 `{ key: value }` と契約を照合
- 型不一致 → **`prop_type_mismatch`** warning + `undefined`
- 必須欠落 → **`prop_missing`** warning + `undefined`
- 未知キー → **`prop_unknown`** warning + 無視

#### テスト: 正常系

- 全 prop 型一致 → 値そのまま
- `default` 適用（未指定時）
- 余分なキー無し

#### テスト: 異常系

- `type: number` に string 渡し → warning + undefined
- `isRequired: true` 欠落 → warning + undefined
- 未知キー `extra` → warning、結果に含めない
- 空契約 + 任意 props → 警告なしで渡す

### renderComponent

- 登録済み alias + props から子ドキュメントを描画
- 手順: loader 取得 → `parsePropsContract` → `validateProps` → `mergeContext` → `renderChildDocument`
- 結果に `\n` があれば **`component_multiline_output`**

#### テスト: 正常系

- `greet({ name: "Ada" })` → 単行 Markdown 文字列
- props 警告があっても描画継続、warnings 集約
- 子の front matter + 親 context が merge される

#### テスト: 異常系

- 子が複数行出力 → **`component_multiline_output`**
- 子ファイル欠落 → **`file_not_found`**
- 未登録 alias → **`parse_error`**
- 子 front matter 不正 → **`parse_error`**

### interpolateExpressions（async 拡張）

- v0.1 同期置換を **async** に変更
- `{{ greet({ name: "Ada" }) }}` で `renderComponent` を呼ぶ
- registry / loader / rootDir / warnings バッグを受け取る

#### テスト: 正常系

- 通常変数 `{{ title }}` は従来どおり
- component 呼び出しが単行に置換
- 同一行に複数 `{{ }}` + component 混在

#### テスト: 異常系

- component 多行出力 → **`component_multiline_output`**
- 式 parse 失敗 → **`parse_error`**
- 危険キー → **`forbidden_property_access`**
- 未登録 component 呼び出し → **`parse_error`**

## Phase 6: パイプライン統合

- `warnings` を include 循環・props 等から集約

- include 子で登録した component を親の式から呼べる
- `{ path }` エントリから component + include 混在ドキュメント

#### テスト: 異常系

- エラー時は warnings まで到達しない（throw）
- registry が子間で共有されないケースが無いこと
