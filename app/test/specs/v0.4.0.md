# v0.4.0 テスト仕様書

[roadmap v0.4.0](../../docs/roadmap.md) の実装範囲に対応する。製品仕様は [docs/specs/](../../docs/specs/) が正。

## スコープ

- **`extend <path>`** によるレイアウト継承（パス解決は include と同じ → [paths.md](../../docs/specs/paths.md)）
- **`block <name>` … `endblock`** によるブロック上書き（Pug / Blade / Twig 風）
- 子テンプレート: **block 以外の本文は無視**
- 未上書き block: layout の **デフォルト本文を残す**
- パイプライン step 3: **extend/block 解決は include より先**（[pipeline.md](../../docs/specs/pipeline.md)）
- component 内 `extend` → **`extend_in_component` 警告 + スキップ**
- `extend` は **先頭 hyogen ブロック必須**（その上の front matter は可）
- v0.3 回帰（281 テスト）維持

## スコープ外（v0.4.0）

- extend の **多重継承**（layout がさらに extend する等）— [templating.md](../../docs/specs/templating.md) 非ゴール
- リモート URL extend — v0.5（loader 拡張後。v0.4 ではパス解決 API のみ同型）
- `@@` ショートハンド — v0.7
- build / SSG 一括 — v0.5

## 確定事項（v0.4.0）

| 項目 | 確定内容 |
|------|----------|
| パイプライン順 | `executeDeclarations` → **`expandExtendsAndBlocks`** → `executeHgBlocks` → `expandIncludes` → `expandControlStructures` → `interpolateExpressions` → `stripHgComments` |
| extend ブロック | **複数行** `@hg`。先頭行 `extend <path>` + 続けて `const` / `let` / 再代入可（[templating.md 例](../../docs/specs/templating.md)）。宣言は step 2 で実行、extend 行は step 3 で消費 |
| block ブロック | **1 行** `@hg`（`block <name>` / `endblock`）。if/each と同型で **本文を挟む** |
| extend 必須位置 | body 内 **最初の** `@hg` ブロックが `extend` で始まること。2 個目以降の extend → **`parse_error`** |
| extend 無し | `expandExtendsAndBlocks` は **no-op**（通常ドキュメントは従来どおり） |
| 子の block 以外 | Markdown 本文・余分な `@hg` は **出力に含めない**（無視） |
| 未上書き block | layout 側 `block … endblock` の **デフォルト本文**をそのまま使用 |
| 複数 block | layout / 子とも **名前で対応**（例: `contents` / `sidebar`）。子が定義しなかった名前は layout デフォルト |
| 同名 block 重複 | 子テンプレート内で **同一 `<name>` を 2 回以上** → **`parse_error`**（後勝ちにしない） |
| layout 内 block 重複 | 同一 layout 内の同名 block も **`parse_error`** |
| context マージ | `mergeContext(parent, layoutFM, childDeclarations)` — **子宣言が後勝ち** |
| 変数の見え方 | マージ後 context で layout 本文・子 block 本文の `{{ }}` を step 6 評価（[variables.md](../../docs/specs/variables.md)） |
| layout 側 extend | layout 本文に `extend` がある場合 → **`parse_error`**（多重継承不可） |
| 循環 extend | `VisitStack` で検出 → **`circular_include` 相当の警告**（code: **`circular_include`**、`via: "extend"`）+ 当該 extend **スキップ**（中断しない） |
| component 内 extend | **`extend_in_component` 警告** + extend 解決 **スキップ**。`block` / `endblock` **指示子のみ除去**、挟まれた本文は **通常 Markdown として残る** |
| `executeHgBlocks` | `block` / `endblock` 1 行ブロックは **pass-through**（step 3 完了後は残らない） |
| 構文エラー | 未対 `block`、余分な `endblock`、`endblock` 欠落、extend 無しで子が `block` のみ → **`parse_error`** |
| 終了タグ | **`endblock` のみ**（`end block` 不可） |
| block 名 | JS 識別子。予約語・空名 → **`parse_error`** |
| strip | step 7 で `extend` / `block` / `endblock` の hyogen コメントを除去 |
| loader | extend 先読込に **必須**（`renderServer({ path })` または注入 loader） |

### レビュー待ち（プレースホルダ）

| 項目 | 候補 / メモ |
|------|-------------|
| extend スキップ時の出力 | 循環・component 内 skip 時、子 block 無しの **layout 全文** vs **空** — v0.4 では **layout 未適用＝extend 前の子 body（block 以外無視済み相当）が空** で固定（循環は警告のみ、markdown は block 未解決のまま step 3 以降へ進まない場合は **throw せず空に近い出力** — 実装時に circular と component の 2 ケースをテストで固定） |

## Phase 1: extend / block パーサ

新規 `app/src/layout/`（または `app/src/extend/`）:

| 関数 | 役割 |
|------|------|
| `parseExtendDirective` | 1 行 `extend <path>` を解析 |
| `parseBlockOpener` | 1 行 `@hg` から `block <name>` / `endblock` を判定 |
| `parseBlockStructures` | ソース走査 → block ツリー（名前・本文範囲） |
| `parseExtendBlock` | 複数行 `@hg` の先頭 extend + 残り宣言行の分離 |

### parseExtendDirective

#### テスト: 正常系

- `extend layout.md` → `{ path: "layout.md" }`
- `extend ./layouts/base.md` / `extend /layouts/base.md`（root あり）
- 前後空白 trim

#### テスト: 異常系

- `extend` のみ（パスなし）→ **`parse_error`**
- `extend layout.md extra` → **`parse_error`**
- `include ./x.md` を渡す → extend ではない（判別可能な結果）

### parseBlockOpener

#### テスト: 正常系

- `block contents` → `{ kind: "block", name: "contents" }`
- `endblock` → `{ kind: "endblock" }`
- `block sidebar_extra`（アンダースコア含む識別子）

#### テスト: 異常系

- `block` のみ（名無し）→ **`parse_error`**
- `end block` → **`parse_error`**
- `block if`（予約語名）→ **`parse_error`**
- 大文字 `BLOCK` → **`parse_error`**（キーワードは小文字想定）

### parseBlockStructures

#### テスト: 正常系

- `block foo` … 本文 … `endblock` 1 組
- 同一ファイル内に **複数名前**の block（layout 想定）
- 空本文（opener と closer が隣接）

#### テスト: 異常系

- `block` 無しで `endblock` → **`parse_error`**
- `endblock` 欠落 → **`parse_error`**
- 同一ファイル内 **同名 block 2 組** → **`parse_error`**
- extend 無しの子ソースに block のみ → **`parse_error`**（孤児 block）

### parseExtendBlock

#### テスト: 正常系

- 先頭 `extend layout.md` + 空行 + `const title = "x"`
- extend のみ 1 行の extend ブロック

#### テスト: 異常系

- 先頭が `const` で extend 無し → extend ブロックではない（別処理）
- 2 行目に `extend` → **`parse_error`**（extend は先頭行のみ）

## Phase 2: レイアウト解決（expandExtendsAndBlocks）

| 関数 | 役割 |
|------|------|
| `expandExtendsAndBlocks` | extend 検出 → layout 読込 → block マージ → merged ソース + context 返却 |

### expandExtendsAndBlocks

#### テスト: 正常系

- **単一継承 + block 上書き**: [templating.md 例](../../docs/specs/templating.md)（`basic/layout.md` + `basic/page.md`）
- **未上書き block デフォルト残存**: `multi-block/` — sidebar は layout デフォルト、contents は子で上書き
- **extend ブロック内宣言**: `title` / `description` が layout の `{{ }}` に反映
- **複数 block 部分上書き**: 子が 1 つのみ上書き、他は layout デフォルト
- extend 無しソース → **入力そのまま**（context 変更なし）
- 解決後ソースに **`block` / `endblock` / `extend` 指示子が残らない**（本文のみの merged Markdown）
- layout 本文中の **通常 Markdown**（`---` 水平線等）が維持される

#### テスト: 異常系

- layout 先 `file_not_found` → **`file_not_found`**（`via: extend`）
- layout 本文に `extend` → **`parse_error`**
- extend が先頭 hyogen でない → **`parse_error`**
- 子で同名 block 重複 → **`parse_error`**
- loader 未指定で extend あり → **Error**（loader required — v0.2 include と同型）

#### テスト: 警告

- **循環 extend** A→B→A → **`circular_include`** warning（`via: "extend"`）+ extend スキップ
- **component 内 extend** → **`extend_in_component`** warning + extend 解決スキップ、block 本文は残存

## Phase 3: 宣言・hg ブロック連携

### executeDeclarations（extend ブロック対応）

#### テスト: 正常系

- extend ブロック内の `const` / `let` を **従来どおり実行**
- 実行後 extend ブロック全体を **除去**（extend 行も除去 — step 3 には extend 情報を別経路で渡すか、step 3 が extend ブロックを直接読む — 実装方針: **step 3 が extend ブロックを scan して処理後除去**。step 2 は extend を含むブロックを **スキップ**（宣言未実行） vs **extend 行を除いて宣言のみ実行** — **後者を採用**: extend 行は残し step 3 へ、宣言は step 2 で実行）

#### テスト: 異常系

- extend ブロック内の不正宣言 → **`parse_error`**（従来どおり）

### executeHgBlocks（block pass-through）

#### テスト: 正常系

- `block contents` / `endblock` 1 行ブロック → **pass-through**（除去しない）
- include / component / if/each は v0.3 どおり

#### テスト: 異常系

- extend 1 行ブロック（`extend layout.md` のみ）→ pass-through または step 3 済みで存在しない — **extend 複数行ブロックのみ**（1 行 extend は **`parse_error`** または複数行ブロック必須 — dsl 例は複数行ブロック内。1 行 `extend` のみも許可）

## Phase 4: パイプライン統合

### renderDocumentBody

```
body
  → executeDeclarations(context)     // extend ブロック内宣言も実行
  → expandExtendsAndBlocks           // NEW step 3

  → expandControlStructures
  → interpolateExpressions
```

#### テスト: 正常系
