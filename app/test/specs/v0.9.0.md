# v0.9.0 テスト仕様書

[roadmap v0.9.0](../../dev-docs/roadmap.md) / [playground.md](../../dev-docs/playground.md) の実装範囲に対応する。  
ライブラリ本体の製品仕様は [dev-docs/specs/](../../dev-docs/specs/) が正。プレイグラウンド製品仕様は [playground.md](../../dev-docs/playground.md) が正。

## スコープ

- 同リポ `playground/`（Vite + Vue + TS + CodeMirror 6）
- **純ロジック**の単体テスト（仮想 FS / loader / src→outDir / 永続化 / Reset）
- `renderClient` + 仮想 FS loader の結合（Node/Vitest 上で可）
- UI（ファイラー見た目・エディタ・プレビューパネル）は **手動確認**（自動テスト対象外）
- v0.8 ライブラリ回帰は維持（playground 追加が `app` の既存テストを壊さないこと）

## スコープ外（v0.9.0）

- UI コンポーネントの自動テスト（Vue マウント等）
- 一括ビルド風の outDir 全展開（当面は開いている 1 ファイルのみ）
- context 編集 UI
- npm 公開・ドキュメントサイト
- データ import / TOC / 許可メソッド追加（v0.10+）

## Phase 1: 仮想 FS

### VirtualFs（想定 API）

パスは POSIX 風（例: `/src/index.md`, `/out/index.md`）。ルートに `src` / `out`（または `outDir`）を持つ。命名は実装で固定しテストでロック。

#### テスト: 正常系

- ファイル作成・読取・更新
- ディレクトリ作成（ネスト可）
- リネーム（ファイル / ディレクトリ）
- 削除（ファイル）。空ディレクトリ削除
- `list` / ツリー列挙でネスト構造が取れる

#### テスト: 異常系

- 存在しないパスの読取 → 明確な失敗（throw または Result。方針をテストで固定）
- ファイルがあるパスへのディレクトリ作成 → 失敗
- `outDir` 配下への **書き込み API を UI 経由で呼ばない**ことは UI 側の責務。ロジック層では「out 領域フラグ」または `writeSrc` / `writeOut` の分離をテストで固定してもよい

## Phase 2: 永続化と Reset

### persist（localStorage 抽象）

テストでは `Storage` をインメモリモックに差し替え可能であること。

#### テスト: 正常系

- FS 状態を serialize → storage に保存 → 新規 FS に hydrate して同一内容
- 「Reset to demo」: FS がシード内容になり、storage もシード相当で上書きされる
- storage が空の初回: シードで初期化

#### テスト: 異常系

- 壊れた JSON / 不正スキーマ → シードにフォールバック（または明示エラー。方針をテストで固定）

---

## Phase 3: デモシード

#### テスト: 正常系

- シードに少なくとも次を含む（パスは実装固定）:
  - エントリ Markdown（`src` 配下）
  - `include` 先
  - `component` 定義
  - `extend` / `block` を使うレイアウトまたはページ
  - `if` / `each` を使う例（同一ファイルでも可）
- シード適用後、エントリを `renderClient` + loader で展開すると **成功**し、期待部分文字列を含む

---

## Phase 4: 仮想 FS loader + render → outDir

### createVirtualLoader(fs)

`renderClient` に渡す `(path) => Promise<string>`。`src` 領域のパスを解決。

#### テスト: 正常系

- include / component が仮想パスで解決される
- 開いているファイルを render し、対応する `outDir` パスに **展開後 markdown** が書かれる
- 警告ありでも markdown が outDir に書かれ、warnings が呼び出し側に返る（または収集される）

#### テスト: 異常系

- 欠落 include → render が reject（またはエラー結果）。**outDir を部分成功で汚さない**（失敗時は前回内容維持 or クリア。方針をテストで固定）
- `outDir` 側パスを loader で読もうとしても src 解決規則に従う（out をソースとして render しない）

### 写像規則（確定してテスト）

例（実装で1つに固定）:

- 入力: `/src/pages/home.md` → 出力: `/out/pages/home.md`（`src` ↔ `out` のプレフィックス置換）

## Phase 5: 診断データの受け渡し（ロジック）

UI は手動だが、render 結果からパネル用に変換する純関数があればテストする。

#### テスト: 正常系

- 成功 + warnings → `{ markdown, warnings }` 形（または同等）
- 失敗（HyogenError）→ `{ error }` 形。markdown なし

#### テスト: 異常系

- （該当すれば）不明な throw → エラーとして扱われる

---

## Phase 6: 統合シナリオ（roadmap）

#### テスト: 正常系

- シード適用 → エントリ render → outDir に出力 → storage 保存 → hydrate 後も outDir / src が復元（out を永続するかは実装方針。少なくとも **src は復元**）
- Reset → シードに戻る

#### 手動確認チェックリスト（自動テストにしない）

- [ ] 左ファイラーでネスト・CRUD
- [ ] エディタ編集でデバウンス後に MD / HTML プレビュー更新
- [ ] エラー時にパネルが目立つこと、警告一覧
- [ ] Reset to demo
- [ ] ローカル `npm run dev`（または同等）で起動できること
