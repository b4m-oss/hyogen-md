# v0.9.1 テスト仕様書

[roadmap v0.9.1](../../docs/roadmap.md) / [playground.md](../../docs/playground.md) の実装範囲に対応する。  
`_` partial のエントリ除外ルールの正は [pipeline.md](../../docs/specs/pipeline.md)（本体の `isUnderscorePartial` と同型）。  
プレイグラウンド製品仕様は [playground.md](../../docs/playground.md) が正。

## スコープ

- Playground の **outDir への `_` 除外**（ファイル名・ディレクトリ名が `_` 始まり）
- `src` には `_` partial を置けること（回帰）
- 開いて **render / プレビュー用 markdown は返す**が、対応 `/out` には書かない
- localStorage 由来の古い `/out/_...` を hydrate 時に掃除
- SRC リネーム直後の **OUT ミラー同期**（子ファイル選択や debounce render を待たない）。`_` → 非 `_` は sync のあと render で out を埋める
- 純ロジックの単体・結合（`playground/test` の Vitest）

## スコープ外（v0.9.1）

- UI 自動テスト
- 一括ビルド風の outDir 全展開
- 出力 Markdown の空行改善（v0.9.2）
- アクションメニュー / `@hg` ハイライト（v0.10.0）
- 公開 API への `isUnderscorePartial` 輸出
- SSG `collectEntryPaths` 本体の変更（既存挙動を再利用するだけ）

## 確定事項（v0.9.1）

| 項目 | 確定内容 |
|------|----------|
| 判定 | パスを `/src` または `/out` から相対化したうえで、いずれかのセグメントが `_` 始まり → 除外対象（本体 `isUnderscorePartial` と同型） |
| `src` | `_` ファイル・`_` ディレクトリを作成・編集・参照可 |
| render | underscore 対象でも `renderClient` は実行し、成功時 `markdown` / `warnings` を返す |
| out 書込 | underscore 対象は **`writeOut` しない**。既に同パスの out があれば **削除** |
| out ツリー | underscore 相当のエントリは存在しない（書かない + hydrate 掃除） |
| 永続化 | `hydrate` / `loadOrSeed` 後に `/out` 配下の underscore エントリを除去してから返す |
| SRC リネーム | リネーム直後に OUT を同期。非 `_`→非 `_` は `/out` ミラーを rename。先が `_` 化なら旧 out を削除。`_`→非 `_` は sync では out を作れないため、ファイルは `scheduleRender`、ディレクトリは配下の非 `_` ファイルを一括 `renderOpenFile`（`renderSrcTreeToOut`）して out を埋める。子選択不要 |
| テスト配置 | `playground/test/**/*.test.ts` |

## モジュール構成（想定）

```
playground/src/fs/
  isUnderscoreEntry.ts        # 仮想パス判定
  persist.ts                  # hydrate 後の掃除（または専用 purge を呼ぶ）

  renderOpenFile.ts           # writeOut 抑制 + stale remove

```

## Phase 1: 判定

### isUnderscoreEntry

- 仮想 FS パス（`/src/...` または `/out/...`）を受け取り、out 除外対象かどうかを boolean で返す
- `/src`・`/out` ルートプレフィックスを剥がした相対パスに対し、セグメントのいずれかが `_` 始まりなら true
- `/src`・`/out` 自身、および通常パスは false。副作用なし

#### テスト: 正常系

- `/src/_meta.md` → true（ファイル名が `_`）
- `/src/_partials/card.md` → true（ディレクトリが `_`）
- `/src/partials/_card.md` → true（ファイル名が `_`）
- `/src/index.md` → false
- `/src/partials/intro.md` → false
- `/out/_meta.md` → true（out 側パスでも同じ規則）
- `/src` および `/out` → false

#### テスト: 異常系

- 空文字 → 明確な失敗（throw。`EINVAL` 相当。`normalizePath` に準拠）
- `/other/a.md`（src/out 外）→ 相対化できなくても、剥がし後のセグメント規則で判定してよい（実装で固定しテストでロック）。推奨: 先頭 `/` を除いた全体をセグメント化し `_` 判定（`other` は false）

## Phase 2: render → out

### renderOpenFile（v0.9.1 追加挙動）

- 従来どおり src ファイルを `renderClient` で展開する
- 成功時、対象が underscore でなければ対応 `/out` へ `writeOut`
- 成功時、対象が underscore なら **out に書かない**。既に `/out` に同相対パスがあれば **remove**
- 失敗時は従来どおり（前回 out を触らない）。underscore 判定は書込前のみ

#### テスト: 正常系

- Given `/src/index.md`（通常）を render → Then `/out/index.md` に展開結果が書かれ、戻り値に `markdown` がある
- Given `/src/_meta.md` を render → Then 戻り値に `markdown` があるが、`/out/_meta.md` は **存在しない**
- Given `/src/_partials/a.md` を render → Then `/out/_partials/a.md` は存在しない（ディレクトリも out に残さない／空親は許容だがファイルは無い）
- Given 先に `/out/_meta.md` を手動 `writeOut` した状態で `/src/_meta.md` を render 成功 → Then `/out/_meta.md` は **削除されている**
- Given `/src/partials/_card.md` を render → Then `/out/partials/_card.md` は存在しない

#### テスト: 異常系

- Given src 外パス → 従来どおり error（out は触らない）
- Given underscore ファイルの render が失敗（例: 欠落 include）→ error を返し、事前に置いた `/out/_meta.md` は **削除しない**（失敗時は out 非変更の既存方針を維持）
- Given 存在しない src ファイル → error（従来どおり）

## Phase 3: 永続化時の掃除

### purgeUnderscoreOutEntries（名前は実装で固定）

- `VirtualFs` 上の `/out` 配下エントリのうち、`isUnderscoreEntry` が true のものを削除する
- ファイルを消した結果、空になった親ディレクトリの扱いは実装任せ（最低限: underscore **ファイル**が残らなければよい）
- `hydrate` および `loadOrSeed` の成功パスから呼ぶ（Reset to demo 後の FS にも余分な `_` out が無いこと）

#### テスト: 正常系

- Given スナップショットに `/src/_meta.md` と `/out/_meta.md` がある → When `hydrate` → Then `/src/_meta.md` は残り、`/out/_meta.md` は無い
- Given `/out/_partials/x.md` と通常の `/out/index.md` がある → When purge → Then `_partials` 側は消え、`/out/index.md` は残る
- Given underscore out が無い通常スナップショット → When `hydrate` → Then 内容は従来どおり（破壊しない）
- Given `loadOrSeed` が不正 JSON でシードフォールバック → Then シード後の out に `_` エントリは無い

#### テスト: 異常系

- Given 壊れた JSON → 既存方針どおりシードフォールバック（v0.9.0）。フォールバック後も underscore out は無い
- Given 不正スキーマ → 同上

## Phase 4: SRC リネーム後の OUT 同期

### syncOutAfterSrcRename

- SRC 側の rename（`fromSrc` → `toSrc`）の **あと**に呼ぶ。パス文字列から `srcToOut` した `fromOut` / `toOut` で OUT を同期する
- `toSrc` が underscore → `fromOut` が存在すれば削除（ツリーごと。空親も掃除してよい）
- `fromSrc` / `toSrc` とも非 underscore かつ `fromOut` が存在 → `fs.rename(fromOut, toOut)` でミラー移動
- `fromSrc` が underscore で `toSrc` が非 underscore → OUT に移すもの無し（何もしない）。呼び出し側が render で out を埋める（下記）
- 子ファイルの選択や debounce を前提にしない
- 副作用は VirtualFs 上の out のみ（localStorage は呼び出し側の `persist`）

### renderSrcTreeToOut（`_` → 非 `_` の補完）

- `syncOutAfterSrcRename` が no-op になる underscore → 非 underscore のあと、OUT を埋めるために使う
- `rootSrcPath`（ファイルまたはディレクトリ）配下の **非 underscore な src ファイル** それぞれに `renderOpenFile` を呼ぶ（underscore パスはスキップ）
- ファイル単位の debounce はしない。呼び出し側が完了後に **一度** `persist` / ツリー更新する
- App: 宛先が **ファイル** なら既存の `scheduleRender`。宛先が **ディレクトリ** なら本ヘルパ

#### テスト: 正常系

- Given `/src/components/badge.md` と `/out/components/badge.md` → When src を `/src/widgets` に rename し sync → Then `/out/widgets/badge.md` があり内容維持、`/out/components` は無い（**追加の select / render なし**）
- Given 同上 → When src を `/src/_components` に rename し sync → Then `/out/components` も `/out/_components` も無い
- Given `/src/a.md` と `/out/a.md` → When `/src/b.md` に rename し sync → Then `/out/b.md` に内容が移り `/out/a.md` は無い
- Given underscore src のみ（out 無し）→ When 非 underscore 名へ rename し sync → Then out は増えない
- Given `/src/_components/badge.md`（out 無し）→ When `/src/components` に rename し sync → Then out はまだ無い。続けて `renderSrcTreeToOut(/src/components)` → Then `/out/components/badge.md` がある（**子 select なし**）
- Given `/src/_meta.md` → When `/src/meta.md` に rename し sync → Then out 無し。続けて `renderOpenFile(/src/meta.md)`（App の `scheduleRender` 相当）→ Then `/out/meta.md` がある

#### テスト: 異常系

- Given `fromOut` が存在しない通常 rename → sync は no-op（throw しない）
- Given `toOut` が既に存在する衝突 → `VirtualFs.rename` の既存方針（`EEXIST`）。呼び出し側で扱う
- Given `renderSrcTreeToOut` に src 外パス → throw（`EINVAL`）
