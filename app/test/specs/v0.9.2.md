# v0.9.2 テスト仕様書

[roadmap v0.9.2](../../dev-docs/roadmap.md) の実装範囲に対応する。  
出力 Markdown の空白・改行アルゴリズムの正は [pipeline.md](../../dev-docs/specs/pipeline.md)。  
ライブラリ本体のみ（Playground の表示補正は対象外）。

## スコープ

- `each` / `if` 展開時に、`preserveHgComments !== true` なら opener/closer の raw コメントを出力に載せない
- `stripHgComments` がコメント除去の **縫い目**で余分な空行を作らない
- 著者ソースに明示された空行（段落間の `\n\n` 等）は尊重する
- 結合: `each` で生成したリストが項目間の空行で切れないこと
- 単体・結合とも `app/test`（Vitest）

## スコープ外（v0.9.2）

- Playground UI / CSS による見た目補正
- ファイル全体の「3 連改行以上を一括圧縮」などのグローバル正規化
- `preserveHgComments: true` のデバッグ出力の見た目改善（現行どおり raw を残す）
- TOC / データ import / Playground UX（他バージョン）

## 確定事項（v0.9.2）

| 項目 | 確定内容 |
|------|----------|
| A. 構造展開 | `preserveHgComments !== true` のとき、`each` は反復ごとに body のみ、`if` は選択 branch の body のみ。opener/closer raw は載せない |
| A'. preserve | `preserveHgComments === true` のときは現行どおり opener/closer raw を残す |
| B. strip 縫い目 | ディレクティブ除去の接合は `max(left, right)` 改行（`joinRemovalSeam`）。最終 strip に加え component / 宣言除去 / block マージでも同じ規則 |
| C. 非対象 | グローバルな `\n\n\n+` → `\n\n` 一括圧縮はしない |
| テスト配置 | `app/test/control/`・`app/test/pipeline/`・必要なら render 結合 |

## Phase 1: 構造展開

### expandEach（preserve オフ）

- `each` … `endeach` を展開し、反復結果の Markdown 本文だけを連結して返す
- `preserveHgComments` が false / 未指定のとき、各反復に `opener.raw` / `closer.raw` を **含めない**
- 副作用: options.warnings 等は既存どおり。本文の `{{ }}` 補間は既存どおり

#### テスト: 正常系

- Given `features = ["a","b","c"]` と本体 `- {{ name }}\n` の each → Then 出力は `- a\n- b\n- c`（または末尾改行付き）。項目間に空行が **無い**。`@hg` / `<!--` が残らない
- Given 空 iterable → Then 空文字（または空白のみ）。コメント残骸無し
- Given 1 要素 → Then body 1 回分のみ

#### テスト: 異常系

- Given nest limit 超過 → 既存どおり警告 + スキップ相当（空寄りの結果）。本バージョンで挙動変更しない
- Given 非 iterable → 既存どおり空。コメント残骸無し（preserve オフ時）

### expandEach（preserve オン）

#### テスト: 正常系

- Given `preserveHgComments: true` → Then 各反復に opener/closer raw（hyogen コメント）が含まれる（現行互換）

### expandIf（preserve オフ）

- 真の branch の body のみを返す。他 branch の opener や endif closer の raw を載せない

#### テスト: 正常系

- Given `if true` と本文 `Yes\n`、else 本文 `No\n` → Then `Yes\n`（または同等）。コメント無し
- Given `if false` / else → Then else body のみ。コメント無し
- Given `if false` のみ（else 無し）→ Then 空寄りの結果。コメント無し

#### テスト: 異常系

- Given nest limit 超過 → 既存方針を維持

### expandIf（preserve オン）

#### テスト: 正常系

- Given `preserveHgComments: true` → Then 現行どおり branch opener / closer raw が残る

## Phase 2: stripHgComments

### stripHgComments（縫い目の畳み）

- 完全な hyogen HTML コメント塊を除去する
- 除去位置で、左右が行境界のときに **除去で増えた空行を 1 段畳む**
- 通常の HTML コメント・未完了 `@hg`・`preserveHgComments: true` は既存どおり

共有ヘルパ `joinRemovalSeam`（`max(left, right)` 改行）は、最終 strip だけでなく **途中段階のディレクティブ除去・置換**にも使う:

- `executeHgBlocks`: `component` ブロック除去
- `executeDeclarations`: 宣言ブロック除去
- `expandExtendsAndBlocks` / `mergeLayoutWithBlocks`: `block` opener/closer 除去と子 body の接合
- `expandIncludes`: include マーカーを子 Markdown に置換する接合

これにより、layout の H1 直後の空行と、子の `block` 直後〜連続 `@hg` 間の空行が積み上がって「見出しの下に 5〜6 行」になる副作用を防ぐ。著者が見出しと本文のあいだに書いた空行（1 行または明示の複数行）は残す。

#### テスト: 正常系

- Given `Hello\n<!--\n@hg\ninclude ./a.md\n@endhg\n-->\nWorld` → Then `Hello\nWorld`（**空行を挟まない**。旧期待 `Hello\n\nWorld` は破棄）
- Given 同一行 minify 形 `Hello<!--@hg ... @endhg-->World` → Then `HelloWorld`（または既存の同一行除去挙動を維持しテストで固定）
- Given 前後に著者空行: `Hello\n\n<!--@hg ... @endhg-->\n\nWorld`（コメント行の外側に明示の空行）→ Then 段落間の空行は **残る**（`Hello\n\nWorld`）。コメント除去で段落がくっついてはならない
- Given hyogen ブロック無し → 入力そのまま
- Given `preserveHgComments: true` → 入力そのまま
- Given 連続する `component` と `include`（間に著者空行）→ component 除去後も include 前後の空行が **二重化しない**

#### テスト: 異常系

- Given 未完了 `<!-- @hg only` → 触れない（既存）
- Given 通常 HTML コメント `<!-- note -->` → 触れない（既存）
- Given 空文字 → `""`

## Phase 3: 結合（render）

### renderDocument / renderClient（each リスト）

- パイプライン全体（展開 → 補間 → strip）のあと、リストが連続していること
- playground デモ相当（`each` + リスト行 + component 呼び出し）を最小フィクスチャで再現してよい

#### テスト: 正常系

- Given features 配列 3 要素と `- {{ name }}`（または badge 相当）の each → Then 出力 Markdown で 3 つのリスト項目が **空行なし**で連続する
- Given each の前後に通常段落 → Then 段落との間の著者空行は維持され、リスト内部だけ詰まっている
- Given `if` で囲った短い本文 → Then 除去後に本文前後へ不要な空段落が増えない
- Given demo 相当: layout の `# title` + 子 `block` 内の連続 `@hg`（component / include）+ intro 本文 → Then H1 と intro のあいだは **空行高々 1**（layout が H1 直後に空行を持つ場合）。`\n{3,}` の隙間を残さない
- Given 著者が H1 と段落のあいだに空行 2 つを書いた（ディレクティブ無し）→ Then そのまま残る

#### テスト: 異常系

- Given `preserveHgComments: true` の render → Then コメントが残る（空行改善の対象外）。結合は 1 件で足りる
